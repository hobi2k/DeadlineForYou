# DeadlineForYou Blueprint

## 1. 한 줄 정의

`DeadlineForYou`는 마감을 앞두고 도망치려는 프리랜서 번역가를 다시 작업 화면으로 끌어오는 캐릭터형 마감 집행 봇이다.

핵심 캐릭터는 마감 집행관 `締切監督`.

목표는 단 하나다.

- 사용자가 프로젝트를 등록하고
- 타이머를 시작하고
- 실제 작업량을 보고하게 만드는 것

## 2. 제품 방향

### 서비스명

- 제품명: `DeadlineForYou`
- 캐릭터명: `마감 집행관 「締切監督」`

### 포지셔닝

- 번역가 전용 압박 코치
- 세션 기반 마감 관리 봇
- 작업 시작과 보고를 강제하는 감독관

### 핵심 가치

- 긴 상담보다 지금 당장 할 일을 준다.
- 프로젝트 상태와 세션 기록을 함께 본다.
- 보고가 들어올 때까지 흐름을 끊지 않는다.
- 필요하면 번역과 이미지 생성도 같은 채널에서 처리한다.

## 3. 타깃 사용자

- 프리랜서 번역가
- 시작 저항이 큰 사람
- 마감 직전까지 일을 미루는 사람
- 작업량은 알고 있지만 착수를 못 하는 사람

언어 조합은 고정하지 않는다.

- `jp -> ko`
- `en -> ko`
- `ch -> ko`
- `ko -> jp`

지원 언어 코드는 `ko`, `jp`, `en`, `ch`로 제한한다.  
프로젝트 등록 시 직접 정하게 설계한다.

## 4. 캐릭터 설계

### Role

마감 집행관 `締切監督`

### 성격

- 냉정
- 직설
- 약간의 조롱
- 하지만 행동 유도에는 진심

### 말투 원칙

- 짧고 단단하게 말한다.
- 변명에 끌려가지 않는다.
- 모욕하지 않는다.
- 답변 끝에는 행동 지시가 있어야 한다.

### 예시

- “그래서 지금 번역은 했어?”
- “하기 싫은 거? 당연하지. 그래도 마감은 온다.”
- “도망치는 속도는 빠른데 번역 속도는 왜 그렇게 느릴까?”

## 5. 대화 운영 원칙

가능하면 모든 응답은 이 순서를 따른다.

1. 현실 코멘트
2. 현재 상태 분석
3. 지금 할 행동
4. 보고 요청

예:

> 지금 또 미루는 중이지?
>
> 마감은 네 감정 기다려주지 않는다.
>
> 지금 할 일:
> 파일 열기
> 다음 문장 번역
> `/timer 15`
>
> 끝나면 `/report <작업량>`으로 보내.

## 6. 핵심 제품 기능

### 6.1 프로젝트 관리

프로젝트별로 저장하는 것:

- 제목
- 원문 언어
- 목표 언어
- 총량
- 완료량
- 마감 시각
- 단위

이제 프로젝트는 단순 숫자 하나가 아니라 파일 묶음 단위로도 관리한다.

추가로 저장하는 것:

- 프로젝트에 연결된 번역 파일 목록
- 파일별 마감
- 파일별 상태
- 어떤 파일이 아직 미완료인지
- 어떤 파일이 마감보다 밀렸는지

프로젝트 운영 액션:

- 등록
- 수정
- 삭제
- 활성 프로젝트 전환

사용자는 총량을 정확히 모를 때 `auto`로 프로젝트를 먼저 등록할 수 있다.  
이후 파일을 올리면 문장 수, 글자 수, 세그먼트 수를 자동 계산해 프로젝트 총량을 다시 잡는다.

### 6.2 파일 기반 작업 진척 추적

파일 단위로 관리하는 이유:

- 실제로 무엇이 남았는지 보이게 하기 위해
- 프로젝트 하나가 여러 파일로 쪼개져 있을 때 병목 파일을 찾기 위해
- “얼마나 남았는가”를 숫자만이 아니라 실제 파일 기준으로 보이게 하기 위해

파일별로 보여줄 것:

- 파일명
- 원문 글자 수
- 원문 세그먼트 수
- 번역 완료 글자 수
- 번역 완료 세그먼트 수
- 파일 상태
- 파일별 마감

상태 예시:

- `pending`
- `in_progress`
- `completed`
- `delayed`

### 6.3 자동 작업량 계산

사용자가 수동으로 총량을 세는 대신 시스템이 자동 계산한다.

자동 계산 대상:

- 글자 수
- 줄 수
- 세그먼트 수
- 남은 파일 수
- 밀린 파일 수

이 집계는:

- `/status`
- 프로젝트 목록
- API overview
- 마감 플래너

에 공통으로 들어간다.

### 6.4 마감 역산 플래너

프로젝트 상태를 읽고 남은 기간 기준 최소 페이스를 계산한다.

예:

- 남은 3일
- 남은 8,000자
- 하루 최소 2,667자 필요
- 시간당 최소 333자 필요

플래너는 단순 정보가 아니라 행동 지시로 연결한다.

예:

- 오늘 최소 몇 세그먼트를 끝내야 하는지
- 지금 25분 타이머에서 몇 단위를 목표로 잡아야 하는지
- 밀린 파일부터 먼저 쳐야 하는지

### 6.5 세션 관리

세션 시작은 `/timer <분>` 하나로 통일한다.

세션 동작:

1. 지정한 분 수로 세션 시작
2. 세션이 10분 이상이면 10분마다 압박 메시지 전송
3. 세션 종료 시 `/report <작업량>` 보고 요청
4. 사용자가 제때 보고하지 않으면 세션을 `0`으로 자동 마감
5. 자동 마감 뒤에는 새 `/timer`를 다시 시작하라고 재촉
6. 보고가 들어오면 진행량 반영과 다음 지시 생성

### 6.6 상태 기반 코칭

입력:

- 하기 싫다
- 졸리다
- 절반도 못 했다
- 막혔다
- 아래 이미지 생성해 줘
- 이 문장 번역해 줘

출력:

- 캐릭터 말투의 코칭 답변
- 권장 타이머
- 즉시 할 일
- 보고 지시

중요:

- 회피 판단은 규칙 기반이 아니라 LLM이 직접 한다.
- 코드는 프로젝트 상태와 세션 흐름만 구조적으로 관리한다.

### 6.7 강한 리마인더 시스템

리마인더는 “끝나면 보고해” 한 번으로 끝나지 않는다.

구조:

1. 세션이 10분 이상이면 10분마다 중간 압박
2. 세션 종료 시 보고 요청
3. 보고가 없으면 `0`으로 자동 마감
4. 자동 마감 뒤에는 새 타이머를 다시 켜라고 재촉
5. 특정 시간대에는 자동 체크인 푸시 전송

자동 체크인 목적:

- 하루를 통째로 날리는 패턴 방지
- 오전, 오후, 밤 체크포인트 확보
- 프로젝트가 있는데도 세션이 없는 상태를 깨기

### 6.8 번역 도구

역할:

- 짧은 문장 번역
- 텔레그램과 API에서 즉시 요청 처리

현재 방향:

- 번역 전용 로컬 모델 사용
- 코칭 모델과 분리
- lazy loading 적용
- 일반 대화에서는 LLM이 tool 호출 여부를 판단
- 업로드된 파일 일부를 잘라 번역 보조 가능

### 6.9 파일 업로드 후 번역 보조

짧은 문장 번역만으로는 실무 가치가 약하다.  
그래서 텍스트 파일을 프로젝트에 바로 붙이고 일부 구간을 번역 보조할 수 있게 한다.

초기 범위:

- `.txt` 파일 업로드
- 업로드 시 자동 작업량 계산
- 파일 ID 발급
- `/file_assist <파일ID> | <원문언어> | <목표언어>` 로 일부 번역 초안 생성

장기 확장:

- 여러 파일 포맷 지원
- 파일 일부 재번역
- 용어집/스타일가이드 반영
- 미완료 구간 자동 탐지

### 6.10 이미지 생성 도구

역할:

- 포스터
- 간단한 콘셉트 이미지
- 챗 안에서 바로 확인 가능한 이미지 생성

현재 방향:

- 텍스트 LLM과 별도 이미지 모델 사용
- 무거우므로 lazy loading과 unload 적용
- 텔레그램 자연어 요청에서도 실제 이미지를 업로드
- 일반 대화에서는 LLM이 tool 호출 여부를 판단

## 7. 인터페이스 설계

### 7.1 API

역할:

- 기능 검증
- Swagger UI 테스트
- 추후 다른 UI 연결의 기반

### 7.2 Telegram bot

역할:

- 실제 사용자 채널
- 프로젝트 등록
- 파일 업로드
- 세션 시작과 보고
- 압박 메시지 수신
- 짧은 번역 / 이미지 요청
- 파일 일부 번역 보조

버튼 원칙:

- 바로 실행 가능한 기능은 명령 버튼으로 둔다.
- 인자가 필요한 기능은 예시를 보여주는 안내 버튼으로 둔다.
- 자세한 설명은 `/help`에 모은다.

현재 버튼:

1. `프로젝트 등록 양식`
2. `프로젝트 수정 양식`
3. `프로젝트 삭제 양식`
4. `프로젝트 목록`
5. `프로젝트 전환 양식`
6. `현재 상태`
7. `타이머 시작 양식`
8. `작업 보고 안내`
9. `번역 양식`
10. `이미지 양식`
11. `파일 번역 보조 양식`
12. `도움말`

프로젝트 등록은 더 쉽게 간다.

- `/deadline_add ...` 명령으로 등록 가능
- 예시 형식 그대로 일반 텍스트를 보내도 등록 가능

즉, 사용자는 명령을 외우지 않아도 된다.

파일 업로드는 버튼 없이도 가능하다.

- 활성 프로젝트가 있으면 `.txt` 파일을 바로 올린다.
- 시스템이 파일을 프로젝트 파일로 등록한다.
- 자동 집계와 파일 ID를 바로 알려준다.

## 8. 기술 방향

### 기본 스택

- Python
- FastAPI
- python-telegram-bot
- SQLite
- transformers
- diffusers

### 모델 방향

- 코칭: `saya_rp_4b_v3`
- 대체 코칭: `Qwen3-4B-Instruct-2507`
- 번역: `rosetta_4b`
- 이미지: `sdxl_turbo`

### 메모리 전략

- 코칭 모델은 상주
- 번역 모델은 lazy loading
- 이미지 모델은 lazy loading + unload
- 이미지 생성 전에 번역 모델을 내릴 수 있게 구성

## 9. 제품 운영 흐름

기본 사용 흐름:

1. 사용자 등록
2. 프로젝트 등록
3. 필요하면 텍스트 파일 업로드
4. `/status`로 현재 상태와 남은 파일 확인
5. `/timer 15` 또는 `/timer 25` 시작
6. 10분마다 압박 메시지 수신
7. 끝날 때까지 보고가 없으면 세션 `0` 자동 마감
8. 새 타이머 시작 재촉 수신
9. 또는 제때 `/report <작업량>` 전송
10. 필요 시 `/translate`, `/image`, `/file_assist` 사용

## 10. 설계 원칙

- 레거시 모드 이름을 사용자에게 노출하지 않는다.
- 사용자가 외워야 할 명령 수를 줄인다.
- LLM이 감정과 저항을 직접 해석한다.
- 코드는 세션과 프로젝트 상태 관리에 집중한다.
- 보고 루프를 끊지 않는 것이 핵심 UX다.

## 11. 재미있는 확장

### 11.1 도장 깨기 미션

세션을 미션처럼 제시한다.

- 첫 문장 돌파
- 첫 페이지 완료
- 용어 통일 점검
- 오탈자 수색

완수 시:

- EXP
- streak
- 배지

### 11.2 냉혹한 통계 리포트

하루 종료 시 정리:

- 오늘 총 집중 시간
- 오늘 완료량
- 세션 수
- 미보고 횟수

목표는 감상문이 아니라 `숫자로 피할 수 없게 만드는 것`이다.
